# F-s27-2: --faint token fails WCAG AA 4.5:1 on every surface it's used against

- Severity: medium
- Status: Verified-Fixed
- Area: a11y

## Issue

The mockup's `#5A6478` ("faint" tertiary text tone), carried into the new
`--faint` token, fails the WCAG 2.1 AA 4.5:1 normal-text contrast requirement
against every real background it lands on in this app: `--bg` (3.25:1),
`--surface-shell` (3.17:1 — the nav section titles and header breadcrumb path),
`--surface` (3.10:1 — table column headers), and `--surface-2` (2.86:1, the
worst case). All four real usages (nav section titles, header breadcrumb,
`data-table th`, the login footnote) are body-adjacent informational text, not
decorative, so this is a real SC 1.4.3 violation, not a corner case.

Found by the T5 a11y auditor via a relative-luminance contrast script (not
eyeballed).

## Resolution — Verified-Fixed

Lightened `--faint` to `#7E88A0` (same blue-gray hue family, lightness raised
only) — `src/index.css`. This clears 4.5:1 on every background the token is
used against (4.80:1 on the worst case `--surface-2`, up to 5.44:1 on `--bg`).
Per doctrine §1.1-E/§2.5 ("when the mockup and accessibility conflict,
accessibility wins and you note the deviation") — documented inline in
`index.css` and in RESULT.md §6.

All other palette pairings (text/muted/badges/buttons/focus ring/links) were
verified passing with margin by the same script; no other token needed
adjustment.
