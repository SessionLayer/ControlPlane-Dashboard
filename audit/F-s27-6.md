# F-s27-6: no 409-conflict unit coverage on 5 Access-config screens

- Severity: medium
- Status: Verified-Fixed
- Area: parity

## Issue

`Rules`/`Roles`/`RoleBindings`/`CertificateAuthorities`/`ServiceAccounts` all
implement the same optimistic-concurrency 409 handling (`access/common.tsx`'s
`MutationError` conflict hint), but none of their test files asserted it —
despite the pattern being tested elsewhere (e.g.
`SessionLimitPoliciesScreen.test.tsx`). A regression to the shared conflict
path would have gone undetected on these 5 screens.

Found by the T5 design-fidelity + parity reviewer.

## Resolution — Verified-Fixed

Added a "surfaces a 409 stale-version conflict with a reload hint" test to
each of the 5 screens' test files, mocking the real `PUT` endpoint returning
409 and asserting both the RFC 9457 title and the "changed since you loaded
it" reload hint render. All 5 pass (53/53 tests across the `access/` suite).
