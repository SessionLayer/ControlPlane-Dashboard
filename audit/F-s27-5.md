# F-s27-5: missing e2e coverage for IA navigation and a real 409 conflict

- Severity: medium
- Status: Verified-Fixed
- Area: parity

## Issue

SESSION.md §5 requires Playwright e2e coverage for "navigate the new IA" and "a
create/edit flow with a 409", beyond the pre-existing login/CSP/replay specs.
Only 4 e2e tests existed (2 CSP, 2 smoke — login and OIDC flow); no test
traversed the rebuilt nav IA, and none exercised a real network-level 409.

Found by the T5 design-fidelity + parity reviewer.

## Resolution — Verified-Fixed

New `e2e/ia-navigation.spec.ts`:

1. Signs in, then clicks one representative nav link per SESSION.md §1.1-B
   group (19 destinations total) and asserts the destination screen's real
   `<h1>` heading renders — a dead link or a throwing screen fails this, it
   doesn't just navigate silently.
2. Mocks a real `PUT /v1/rules/:id` 409 response and drives the full
   click-path (open row → Edit → Save) through the actual built app, asserting
   the conflict title and reload hint render — a genuine network-level
   assertion, distinct from the MSW-mocked unit tests.

Both pass against the production build (`npm run test:e2e`), alongside the
pre-existing 4 specs (6/6 green).
