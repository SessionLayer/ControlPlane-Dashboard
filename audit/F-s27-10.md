# F-s27-10: RoleBindingsScreen shows the raw role UUID, not a resolved name

- Severity: low
- Status: Accepted-Risk
- Area: parity

## Issue

`RoleBindingsScreen.tsx`'s list shows the binding's `roleId` as a raw UUID
under the honest label "Role ID"; the mockup shows a resolved role name.
Pre-existing on `main`, not introduced by this session.

Found by the T5 design-fidelity + parity reviewer.

## Disposition — Accepted-Risk

Resolving the name would require a client-side join against `useRoles()`, but
that hook is a cursor-paginated list (`useCursorList`) — only the first page
is loaded until "Load more" is clicked, so a join could only resolve names for
roles on the currently-loaded page and would silently fall back to the ID for
the rest. That's a worse UX than the current honest, unambiguous "Role ID"
label (partial, unpredictable resolution reads as a bug, not a feature). No
batch role-lookup-by-id endpoint exists in the contract to do this properly,
and adding one is a contract change (S28's territory, not this session's).
Per doctrine ("derived, never invented") the current display is the correct
choice until such an endpoint exists — showing a real, complete field beats
showing a sometimes-resolved, sometimes-not one.
