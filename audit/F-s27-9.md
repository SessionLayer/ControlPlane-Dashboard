# F-s27-9: Idempotency-Key OBS documentation was inconsistent across mutations sharing the same contract gap

- Severity: low
- Status: Verified-Fixed
- Area: parity

## Issue

Several write operations have no contract-defined `Idempotency-Key` parameter
(only the S17 config-CRUD resources do) — a genuine, frozen-contract gap, not
a Dashboard defect. `src/features/ir/hooks.ts` documented this inline ("OBS-1")
at JIT submit, FIDO2 register, offline-code issue, and OTP issue — but
`useRegisterNode` (`fleet/api.ts`) and `useCreateLock` (`ir/hooks.ts`) have the
identical gap with no matching comment, so the documentation looked spottier
than the actual (consistent) contract behavior.

Found by the T5 design-fidelity + parity reviewer.

## Resolution — Verified-Fixed

Added the same "NB: no contract-defined Idempotency-Key parameter... OBS-1"
comment to `useRegisterNode` and `useCreateLock`, so every affected mutation is
documented the same way. No code/contract change (out of scope this session —
S28 owns contract changes); see RESULT.md §7 for the consolidated OBS list.
