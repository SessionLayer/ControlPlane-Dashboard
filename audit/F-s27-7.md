# F-s27-7: service-account credential revoke had no confirm dialog

- Severity: medium
- Status: Verified-Fixed
- Area: parity

## Issue

`ServiceAccountsScreen.tsx`'s `CredentialsPanel` called `revoke.mutate(...)`
directly from both revoke buttons ("Revoke this credential" and "Revoke by
ID") with no confirmation step — the only destructive action in the entire
Access-config surface without one (every delete/revoke/rotate elsewhere in the
app sits behind `ConfirmDialog`). Revocation denies new sessions immediately
and cannot be undone; an operator could trigger it by a single misclick.
Pre-existing on `main`, but this session touched the file (adding the inline
Rotate/OTP-style affordances elsewhere in Access-config), so it was in scope to
fix rather than carry forward.

Found by the T5 design-fidelity + parity reviewer.

## Resolution — Verified-Fixed

Both revoke buttons now set a `pendingRevoke` id instead of mutating directly;
a `ConfirmDialog` ("Revoke this credential?") gates the actual mutation,
matching every other destructive action's pattern in the app. Regression:
`ServiceAccountsScreen.test.tsx` — confirms revoke fires only after the dialog
is confirmed, and that Cancel leaves the credential un-revoked.
