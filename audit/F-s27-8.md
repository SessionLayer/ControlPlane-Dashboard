# F-s27-8: JIT policies' approval-chain column and service-accounts' Token TTL column were thinner than the mockup and schema support

- Severity: low
- Status: Verified-Fixed
- Area: parity

## Issue

Two small design-fidelity gaps found in the same review pass:

1. `JitPoliciesScreen.tsx` showed only an approval-chain length (`r.approvalChain.length`) where the mockup shows a descriptive summary ("2 levels (dba → security)"); the full chain required opening the detail dialog.
2. `ServiceAccountsScreen.tsx`'s list omitted the schema-backed `tokenTtlSeconds`
   column both the mockup and `ServiceAccountResource` support; it was visible
   only in the detail dialog.

Found by the T5 design-fidelity + parity reviewer.

## Resolution — Verified-Fixed

Added `ApprovalChainSummary` to `policies/common.tsx` (mirrors the existing
`SelectorSummary` pattern) — renders "N levels (kind:value → kind:value)" or
"auto-approve (0 levels)", derived entirely from real `JitApprovalLevel` data,
nothing invented. `JitPoliciesScreen.tsx`'s "Approvals" column now uses it.
`ServiceAccountsScreen.tsx` gained a right-aligned "Token TTL (s)" column
reading `r.tokenTtlSeconds` (em-dash when absent). Both covered by their
screens' existing test suites (no new states introduced beyond what was
already exercised).
